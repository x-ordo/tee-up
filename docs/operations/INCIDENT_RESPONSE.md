# INCIDENT_RESPONSE.md

## 인시던트 대응 가이드

> **본 문서는 서비스 장애 발생 시 대응 절차를 정의합니다.**

---

## 1. 인시던트 심각도 분류

| 레벨 | 이름 | 설명 | 응답 시간 | 예시 |
|------|------|------|----------|------|
| **P0** | Critical | 전체 서비스 장애 | 15분 | 사이트 다운, 데이터 손실 |
| **P1** | High | 주요 기능 장애 | 30분 | 로그인 불가, 결제 실패 |
| **P2** | Medium | 부분 기능 장애 | 2시간 | 일부 페이지 오류, 느린 응답 |
| **P3** | Low | 경미한 이슈 | 24시간 | UI 버그, 오타 |

---

## 2. 인시던트 대응 프로세스

```
┌─────────────────────────────────────────────────────────────┐
│                    인시던트 대응 플로우                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 감지 ──▶ 2. 분류 ──▶ 3. 대응 ──▶ 4. 해결 ──▶ 5. 검토    │
│     │         │          │          │          │           │
│  알림 수신   심각도 판단  팀 소집    문제 해결   포스트모템   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 감지 (Detection)

**알림 채널:**
- Sentry 에러 알림
- UptimeRobot 다운타임 알림
- Slack #teeup-alerts 채널
- PagerDuty (P0/P1)

### 2.2 분류 (Triage)

1. 심각도 판단
2. 영향 범위 파악
3. 인시던트 커맨더 지정

### 2.3 대응 (Response)

**P0/P1:**
```bash
# 1. 인시던트 채널 생성
# Slack: #incident-YYYY-MM-DD-title

# 2. 팀 소집
# @channel 인시던트 발생!

# 3. 상태 페이지 업데이트
# StatusPage 업데이트

# 4. 롤백 고려
```

### 2.4 해결 (Resolution)

1. 원인 파악
2. 수정 적용
3. 모니터링 확인
4. 서비스 정상화 확인

### 2.5 검토 (Review)

- 포스트모템 작성 (72시간 이내)
- 재발 방지 대책 수립
- 프로세스 개선

## 3. 역할 및 책임

| 역할 | 책임 |
|------|------|
| **인시던트 커맨더** | 전체 대응 조율, 의사결정 |
| **테크 리드** | 기술적 문제 해결 |
| **커뮤니케이션** | 내/외부 커뮤니케이션 |
| **서기** | 타임라인 기록 |

## 4. 커뮤니케이션 템플릿

### 4.1 내부 알림 (Slack)

```markdown
🚨 **인시던트 발생**

**심각도:** P1 - High
**상태:** 조사 중
**영향:** 사용자 로그인 불가
**시작 시간:** 2025-11-25 14:30 KST
**담당자:** @tech-lead

**현재 상황:**
- 로그인 API 500 에러 발생
- 원인 조사 중

**다음 업데이트:** 15분 후
```

### 4.2 상태 페이지

```markdown
**인시던트 - 로그인 서비스 장애**

**상태:** 조사 중
**영향 서비스:** 로그인, 회원가입

**업데이트:**
- 14:30 - 문제 감지, 조사 시작
- 14:45 - 원인 파악, 수정 중
- 15:00 - 서비스 복구 완료
```
